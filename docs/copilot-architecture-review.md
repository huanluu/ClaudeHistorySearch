## 1. Health-check contract is broken between client and server

**Severity:** Critical

**Problem:** The client only accepts `/health` when the JSON payload contains `status == "ok"` in `Shared/Sources/ClaudeHistoryShared/Services/ServerDiscovery.swift:71-82`. The server returns `status: "healthy"` from `server/src/features/admin/routes.ts:53-58`, and `server/src/features/admin/DiagnosticsService.ts:85-91` also emits `"healthy"`. The server tests lock that in at `server/src/features/admin/routes.test.ts:31-54`. The iOS app depends on this path in `ClaudeHistorySearch/App/ClaudeHistorySearchApp.swift:19-34`.

**Why it matters:** Cached URL verification is guaranteed to fail, so the app falls back to Bonjour/manual/localhost even when the server is healthy. On corporate networks, where Bonjour is often unreliable, this becomes a real startup failure, not a cosmetic bug. It also escaped detection because server tests and client tests validate different contracts; `Shared/Tests/ClaudeHistorySharedTests/ServerDiscoveryTests.swift:22-185` never exercises the real `/health` payload.

**Proposed fix:** Pick one health contract and make it shared. Standardize on `status: "healthy"` plus optional checks. Create a shared `HealthResponse` model in the Swift package and a matching TS DTO, update `ServerDiscovery.verifyServerHealth()` to decode that model instead of raw JSON, and add one contract test that hits the actual server `/health` response and parses it with the Swift-side logic. Stop using magic status strings in two codebases.

## 2. Authentication and transport security are weak by default

**Severity:** Critical

**Problem:** `server/src/shared/provider/auth/middleware.ts:8-30` skips auth entirely when no API key is configured and permanently leaves `/admin` public. The WebSocket client sends the API key in the URL query string at `Shared/Sources/ClaudeHistoryShared/Services/WebSocketClient.swift:235-242`, and the server only validates query-string auth in `server/src/gateway/WebSocketGateway.ts:164-176`. On top of that, `server/src/gateway/HttpTransport.ts:39-43` enables `Access-Control-Allow-Origin: *` while allowing `X-API-Key`.

**Why it matters:** First-run means open access to the server on the local network. `/admin` exposes diagnostics/config UI without auth. Query-string API keys leak into logs, proxies, browser history, and network tooling. Wildcard CORS plus header auth broadens the attack surface further. This is exactly the wrong trust model for a tool running on a corporate machine and reachable over LAN/tunnels.

**Proposed fix:** Make auth required by default. If you want a bootstrap mode, bind it to `localhost` only and make it explicit. Remove `'/admin'` from `PUBLIC_PATHS`. Move WebSocket auth to a header-based `URLRequest` (`URLSession.webSocketTask(with: URLRequest)`) or a first-message auth frame; stop accepting `apiKey` in the URL. Tighten CORS to an explicit allowlist or disable it entirely unless a browser client genuinely needs it. Add end-to-end tests for "no key configured," "admin route," and WebSocket auth transport.

## 3. Heartbeat work-item execution uses unsafe shell-string subprocesses

**Severity:** High

**Problem:** `server/src/features/scheduler/workItems.ts:8-13` builds a shell command string with interpolated WIQL and passes it to `executor.execSync(...)`. The adapter behind that abstraction, `server/src/shared/infra/runtime/NodeCommandExecutor.ts:1-12`, calls raw `child_process.execSync(command)`. That means the architectural seam is a shell-string API, not a safe subprocess API.

**Why it matters:** Today the WIQL is static, so the immediate exploitability is low. The real problem is the abstraction: the minute any variable input touches that path, it becomes command-injection-prone. It also makes quoting and escaping brittle and undermines the project's own subprocess-safety claims. This is technical debt disguised as an adapter.

**Proposed fix:** Delete the stringly-typed executor contract. Replace it with an arg-array interface such as `run(command: string, args: string[], options)` and implement it with `spawnSync` or `execFileSync`. Move Azure DevOps query execution into a dedicated infra adapter with a typed request/response API. Pass WIQL as an argument, never as shell-interpolated text. Add tests that assert exact argv construction and fail if any feature tries to pass a full shell command.

## 4. The Swift shared layer creates duplicate service instances and collapses service/UI boundaries

**Severity:** High

**Problem:** `Shared/Sources/ClaudeHistoryShared/Services/APIClient.swift:17-48` makes the network client an `ObservableObject` with UI state, and `Shared/Sources/ClaudeHistoryShared/Services/NetworkService.swift:7-18` bakes `@MainActor` and UI-facing state into the protocol itself. Meanwhile, views create fresh clients internally instead of using the app-owned instances: `ClaudeHistorySearch/Views/SessionListView.swift:21-24`, `ClaudeHistorySearchMac/Views/SearchPopoverView.swift:27-29`, and `Shared/Sources/ClaudeHistoryShared/Views/SessionView.swift:47-50`, `62-65`, `86-89`.

**Why it matters:** You now have hidden, divergent connection/auth state. One `APIClient` can have the right base URL and API key while another, created inside a view, does not. That creates non-deterministic bugs, makes dependency injection fake, and turns the "shared package" into a second application layer instead of a reusable library. It will get worse as Mac-specific workflows expand.

**Proposed fix:** Introduce a single app-owned service container per target and inject those services everywhere. Ban `APIClient()` and `WebSocketClient()` creation inside views. Make services plain classes/actors without `ObservableObject`; move UI observation into view models or a dedicated app store. Keep the shared package focused on models, protocols, and presentation components; push target-specific wiring and workflow orchestration back into the app targets.

## 5. The realtime architecture is single-consumer and cannot support proactive push

**Severity:** High

**Problem:** `Shared/Sources/ClaudeHistoryShared/Services/WebSocketClient.swift:212-216` exposes exactly one `onMessage` callback. `Shared/Sources/ClaudeHistoryShared/ViewModels/ChatViewModel.swift:105-121` explicitly documents that only one consumer can listen at a time, and `Shared/Sources/ClaudeHistoryShared/ViewModels/SessionViewModel.swift:307-315` overwrites the same slot. On the server side, the gateway already exposes `broadcast()` in `server/src/gateway/types.ts:35-39`, but it is effectively unused in production, while `server/src/app.ts:126-129` still installs a generic echo handler.

**Why it matters:** This blocks the stated product direction. You cannot have a durable background listener for proactive push if opening Chat or Session view steals the only message handler. You also have no app-level event bus, no concurrent subscriptions, and no clean path for long-running background tasks to surface into the UI. This is not a polish issue; it is a foundational mismatch with the product vision.

**Proposed fix:** Replace the single callback with a subscription model: typed observers by message type, or an `AsyncSequence`/fan-out coordinator that multiple consumers can subscribe to safely. Add a realtime coordinator/store that owns the socket independently of current screen lifetime. Define explicit push message types in `server/src/gateway/protocol.ts`, remove the echo handler from `app.ts`, and use `broadcast()` or targeted pushes for server-originated events. Then add integration tests with multiple simultaneous subscribers and at least one server-initiated push event.

# Prevention Mechanisms

> How to make each class of bug structurally impossible, not just fixed.
> Generated by Copilot (GPT-5.4) follow-up on 2026-03-19.

The pattern across all 5 issues: **remove the ability to express the bug in the type system or fail CI when someone tries.**

---

## 1. Health contract drift → Shared contract artifact + dual-language contract tests

**Enforcement type:** Contract test + typed model + CI gate

**Mechanism:**
- Create a shared contract artifact (e.g., `contracts/health-response.schema.json` or `.example.json`)
- **Server test**: hits `/health`, validates response against the contract schema
- **Swift test**: decodes the same artifact using the real `HealthResponse` decoder from `ServerDiscovery`
- Replace raw string checks (`json["status"] as? String == "ok"`) with typed decoding:

```swift
public struct HealthResponse: Decodable, Sendable {
    public enum Status: String, Decodable {
        case healthy, degraded, unhealthy
    }
    public let status: Status
    public let timestamp: String
}
```

```ts
const HealthResponseSchema = z.object({
  status: z.enum(['healthy', 'degraded', 'unhealthy']),
  timestamp: z.string(),
  checks: z.object({ database: z.boolean() }).optional(),
});
```

String drift now fails tests in both languages automatically.

## 2. Auth/security regressions → Scorecard invariants + type-level auth mode + source scans

**Enforcement type:** Scorecard security tests + regex source scans + type constraint

**Mechanism:**

**A. No public admin paths** — scorecard test fails if `PUBLIC_PATHS` contains anything except `/health`:
```ts
expect(publicPaths).toEqual(['/health']);
```

**B. No query-string API key auth** — source scan regexes:
```ts
// Server: fail on
/searchParams\.get\(['"]apiKey['"]\)/
```
```swift
// Swift: fail on
/URLQueryItem\(name:\s*"apiKey"/
```

**C. No wildcard CORS with auth headers** — scorecard test fails if a file contains both `Access-Control-Allow-Origin', '*'` and `X-API-Key`

**D. Remove "open when no key" as a representable state** — replace boolean bypass with typed auth mode:
```ts
type AuthMode = 'required' | 'bootstrap-localhost-only';
createAuthMiddleware({ mode, hasApiKey, validateApiKey, isLoopback })
```

The insecure global-open mode no longer exists in the type system.

## 3. Shell-string injection → Unsafe primitive removed from port + lint ban

**Enforcement type:** Type-level constraint + ESLint rule + scorecard invariant

**Mechanism:**

**A. Change the port so shell strings are impossible:**
```ts
// Before (unsafe)
execSync(command: string, options?: object): string

// After (safe)
interface CommandRunner {
  run(command: string, args: readonly string[], options?: {
    cwd?: string;
    env?: Record<string, string>;
  }): { stdout: string; stderr: string; exitCode: number };
}
```

**B. Ban `execSync` in production code** — ESLint `no-restricted-syntax`:
```js
{ selector: "CallExpression[callee.name='execSync']", message: 'execSync is forbidden in src/.' }
{ selector: "ImportSpecifier[imported.name='execSync']", message: 'Use argv-based execution only.' }
```

**C. Scorecard regex** — fail on any `execSync(` in `src/**` with zero-file allowlist.

---

## 4. Duplicate service instances → API redesign + Swift source-architecture tests

**Enforcement type:** Type/API redesign + XCTest source scan + visibility constraints

**Mechanism:**

**A. Views accept view models, not concrete services:**
```swift
// Before (allows rogue construction)
public init(sessionId: String, webSocketClient: WebSocketClient? = nil)

// After (no place to call APIClient())
public init(viewModel: SessionViewModel)
```

**B. Ban `ObservableObject`/`@Published` in service files** — XCTest source scan:
- Fail if any file under `Shared/.../Services/**` contains `: ObservableObject` or `@Published`

**C. Ban service construction in view files** — XCTest source scan fails on:
```regex
\bAPIClient\s*\(
\bWebSocketClient\s*\(
\bServerDiscovery\s*\(
```
in files under `**/Views/**`, with allowlist for app bootstrap files only (`ClaudeHistorySearchApp.swift`, `AppDelegate.swift`).

---

## 5. Single-consumer WebSocket → Protocol redesign + fan-out tests

**Enforcement type:** Type-system constraint + integration tests + source scan

**Mechanism:**

**A. Delete `onMessage` from the protocol** — replace with multicast API:
```swift
// Before (single consumer, last-writer-wins)
var onMessage: ((WSMessage) -> Void)? { get set }

// After (multi-consumer)
func subscribe(_ handler: @escaping @Sendable (WSMessage) -> Void) -> SubscriptionToken
// or
var messages: AsyncStream<WSMessage> { get }
```

Compiler prevents callback reassignment because the API no longer permits it.

**B. Fan-out integration test:**
- Create one WebSocketClient, register two subscribers
- Inject one incoming message → assert both receive it
- Cancel one → assert the other still receives later messages

**C. Ban direct callback reassignment** — Swift source scan fails on `\.onMessage\s*=` outside the transport implementation.

**D. Server push contract test** — fail unless at least one server-originated push type exists in `protocol.ts` beyond the generic `'message'` type.

## Summary

| Issue | Prevention Pattern | Enforcement |
|-------|-------------------|-------------|
| 1. Contract drift | Shared schema + dual-language contract tests | CI (both test suites) |
| 2. Auth regressions | Scorecard invariants + typed auth mode | CI + type system |
| 3. Shell injection | Unsafe API removed from port + lint ban | Type system + ESLint + scorecard |
| 4. Service duplication | Views can't construct services + source scans | API design + XCTest architecture tests |

**The principle:** "We fixed the bug" vs. "this category of bug no longer fits in the architecture."
